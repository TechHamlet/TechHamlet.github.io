How to Set Up Correct File Permission for Your WordPress Blog

Feb 16th, 2012 | Comments

On computer file systems, different directories and files have “permissions” that describe what and who can read, modify, write and access them.  This proves necessary because WordPress might need to write to files in the wp-content directory to activate certain functions.

WordPress and permission schemes

Permissions will determine different depending on the host, so this guide will only involve general principals.  There is no way possible to cover all bases when it comes to permission schemes. They are different from host to host, so this article only details general principles. It cannot cover all cases.

Generally speaking, all files should be owned by the user (ftp) account on your web server, and it should be writable on the account.  Shared host files should never be owned by the web server processes.  A file that needs to write access from WordPress should always be group-owned or individually owned by the user account from WordPress. The folders and files permissions of WordPress should always be the same for nearly all users, depending on the type of installation that was performed and the umask settings of a systems environment when the installation occurs.

Important note

If you installed WordPress by yourself, you probably will not need to modify the file permissions.  If you are having problems with permission errors, or to modify permissions, you should probably not mess with it.

Generally, all core WordPress files should be writable by the httpd account or the user account.  If you use mod_rewrite Permalinks or other .htaccess features it proves vital to ensure that WordPress can write to your /.htaccess file.  If you want the built-in theme editor, all the files need to be grouped as writable.  Before you modify file permissions, try using it first, to see if it works.

Sometimes, plugins need the /wp-content/folder to be made writable but sometimes, they will not let you know during the installation process.  Sometimes, this might involve assigning 755 permissions.  The same applies for /wp-content/cache and /wp-content/uploads/.  Other directories under /wp-content/should always be documented by what the plugin/theme requires.  The permissions will always vary.

Shared hosting with suexec

This approach proves popular and is used by various web hosts that use suexec for running PHP binaries. These systems the php process funs as the owner of the php files themselves, therefore allowing a more secure environment and a simpler configuration for certain cases of shared hosting.

Important note

Suexec should not be used on single-site server configurations.  They are more reliable for certain cases of shared hosting.

 The correct permissions scheme for a suexec configuration proves straightforward.

  • Files should be owned by the existing user account, not the account used for the httpd process.

  • Directories should be 750 or 755

  • Group ownership does not matter, unless there is a particular group requirement for the web server process permissions.

  • Files should be 640 or 644. There is always one exception to the rule, and that is wp-config.php, and it is placed at 600 in order to prevent others on the server from reading it.

  • Directories should never be given 777, not even to upload directories. The reasoning for this is because the php process is running as the file owner and it gains the owners permissions and writes to even a 755 directory.

WordPress will recognize that it can create files with appropriate ownership with this type of setup.  It will not ask for FTP credentials when installing or upgrading plugins.

Unhide hidden files

By default, FTP clients keep hidden files that begin with a period (.), from being seen.  At some point, you might want to see the hidden files, so you can modify the permissions on a file. To show hidden files in FileZilla, it proves necessary to select “View” from the menu at the top.  Choose “Show hidden files”. The screen will display the files and they will refresh, and a hidden file should easily come to view.

Chmod

Chmod is a UNIX command that means “Changed mode” on files. The –R flag means to implement changes to every directory and file inside of wp-content.  766 is the mode that is used in changing the directory to, it simply means that the directory is writable and readable by WordPress and other users on your system. Lastly, if 766 does not work, and then try 777 which make all folders and files readable, executable and writable by all groups, processes and users.

If you are using Permalinks, you should change permissions of .htaccess to ensure that WordPress can upgrade when you change the settings such as redirect, category or adding a new page etc… This requires an update to the .htaccess file when mod-rewrite Permalinks are used.

  1. Go to WordPress main directory

  2. Enter chmod –v 666 .htaccess

Finding a workaround

It proves easy to have the enhanced features of WordPress plugins. This decreases the chances of placing yourself at risk. Contact your server support or the Plugin author to request a workaround.

Conclusion

The .htaccess file is a file that is accessed by the owner. If you set the permissions too low, your server cannot access files and will cause errors. This is why it proves essential to get the most secure settings.  Always begin restrictive with increasing permissions until it works.
